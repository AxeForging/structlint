# Spec 009 — `structlint init --infer`

## Problem

`structlint init` generates a config from one of four fixed templates (`projectTemplates` in `internal/cli/init.go`), picked by `detectProjectType`. For any repo that doesn't look like a textbook Go/Node/Python layout, the very first `validate` after `init` drowns the user in violations — the template's `allowedPaths` and `file_naming_pattern.allowed` describe an idealized project, not *this* project. That's a terrible first five minutes: adopting structlint on an existing codebase means hand-editing the config until the noise stops, which is exactly the busywork the tool exists to remove.

## Approach

Add `--infer` to `init`: instead of a canned template, walk the actual tree (via the `validator.Snapshot`/`*Tree` API introduced in **spec 005** — this spec depends on it) and generate a config that describes what is really there. The generated config is a *faithful baseline*: `init --infer && validate` must exit 0 on the same tree. Tightening (removing entries, adding `disallowed`) is then an intentional, incremental act by the user.

Heuristics (deliberately shallow — a baseline, not a taxonomy):

- **allowedPaths**: `"."` plus one entry per depth-1 directory — `name/**` when the dir has children, bare `name` when empty/leaf.
- **file_naming_pattern.allowed**: one `*.ext` per extension observed anywhere in the tree; extensionless files (`Makefile`, `LICENSE`, `Dockerfile`, dotfiles like `.gitignore`) become exact-name entries.
- **required**: seeded only from certainties present in the tree — `go.mod`, `README.md`. Nothing speculative; a wrong `required` entry makes the primary property fail.
- **disallowed / disallowedPaths**: left empty. Inferring prohibitions from absence is guesswork.
- **ignore**: the default ignore set (`.git`, `node_modules`, `vendor`, `dist`, `build`, `bin`) applied *during* the walk and emitted in the config, so what we skipped stays skipped at validate time.

`--infer` is mutually exclusive with `--type` (error if both). The existing `--force` guard on an existing config file applies unchanged.

## Non-goals

- No placement/boundary/requiredGroups inference — those encode intent a tree walk can't see.
- No "smart" tightening (e.g. guessing `*.env*` should be disallowed). That's `suggest`'s advisory territory (spec 010) and template territory (`--type`).
- No deep glob synthesis beyond depth 1 (`internal/cli/**` style nesting comes free via `name/**`).
- Not changing the default `init` behavior or its templates.

## Backward compatibility

- Purely additive flag; `init`, `init --type`, and auto-detection are byte-identical to today.
- Depends on spec 005's `Snapshot(root, ignore) → *Tree` being merged first.

## Design

### CLI (`internal/cli/init.go`)

```go
&cli.BoolFlag{
    Name:  "infer",
    Usage: "generate config by inspecting the current tree instead of using a template",
},
```

Action: if `cmd.Bool("infer") && cmd.IsSet("type")` → error `--infer and --type are mutually exclusive`. Then `infer.Generate(".")` → YAML bytes → same existing-file/`--force`/write path as templates.

### New package `internal/infer/`

- `infer.go` — `Generate(root string) ([]byte, error)`: calls `validator.Snapshot(root, defaultIgnore)`, folds the `Tree` entries through the heuristics, renders YAML.
- `heuristics.go` — pure functions, each independently testable:
  - `AllowedPaths(t *Tree) []string` — `"."` + sorted depth-1 dirs (`name/**` / bare `name`).
  - `AllowedFilePatterns(t *Tree) []string` — sorted, deduped `*.ext` + exact extensionless names.
  - `RequiredFiles(t *Tree) []string` — subset of `{go.mod, README.md}` present at root.
- Output is rendered from a small template (like `projectTemplates`) with a header comment `# generated by structlint init --infer — a baseline of the current tree; tighten incrementally`, not `yaml.Marshal` of `config.Config`, so ordering and comments are deterministic.

Determinism contract: same tree → byte-identical config (sorted sections, stable template).

## Implementation steps

1. Add `internal/infer/heuristics.go` with the three pure heuristic functions + unit-style coverage via the binary fixtures (see Tests).
2. Add `internal/infer/infer.go` with `Generate` wiring Snapshot → heuristics → YAML template.
3. Wire `--infer` flag + mutual-exclusion check + generation path into `internal/cli/init.go`.
4. Add `test/init_infer_test.go` (fixtures + round-trip property).
5. Update `docs/user/cli-reference.md` and `docs/user/getting-started.md` (adoption flow: `init --infer` → `validate` → tighten).

## Tests

`test/init_infer_test.go` (binary-based via `buildBinary`, per team convention):

- `TestInitInfer_ValidatePassesOnSameTree` — the primary property, table-driven over go-shaped, node-shaped, and mixed/weird fixture trees built in `t.TempDir()`: run `init --infer`, then `validate`, assert exit 0.
- `TestInitInfer_Depth1DirsBecomeGlobs` — tree with `internal/` (has children) and `empty/` (leaf); assert config contains `internal/**` and bare `empty`.
- `TestInitInfer_ExtensionsAndExactNames` — tree with `.go`, `.md`, `Makefile`, `.gitignore`; assert `*.go`, `*.md`, exact `Makefile`, exact `.gitignore` in `allowed`.
- `TestInitInfer_RequiredOnlyFromCertainties` — tree with `go.mod` + `README.md` → both required; tree with neither → `required` empty.
- `TestInitInfer_MutuallyExclusiveWithType` — `init --infer --type go` exits non-zero with the exclusivity message.
- `TestInitInfer_RespectsForceGuard` — existing `.structlint.yaml`, `init --infer` fails without `--force`, succeeds with it.
- `TestInitInfer_Deterministic` — run twice on the same tree, assert byte-identical output.
